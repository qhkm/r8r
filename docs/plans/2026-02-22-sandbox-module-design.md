# Sandbox Module Design

> Pluggable code execution sandbox for r8r workflow engine.
>
> Date: 2026-02-22

## Problem

r8r workflows need to execute arbitrary code (Python, Node, Bash) generated by AI agents or written by workflow authors. The code may be untrusted. r8r needs a sandboxed execution environment that:

- Works on both macOS (dev) and Linux (production)
- Is pluggable so operators can choose isolation level
- Follows r8r's existing patterns (EventBackend trait, feature flags)
- Ships as part of the single binary with no mandatory external deps

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Architecture | Thin trait + feature-flagged backends | Follows existing EventBackend pattern |
| Default backend | Subprocess | Zero deps, works everywhere |
| Production backend | Docker (via bollard SDK) | Good isolation, macOS + Linux |
| Future backend | Firecracker | Strongest isolation, Linux + KVM only |
| Config scope | Global default, runtime auto-resolved | Operators set backend once in r8r.toml |
| v1 output | stdout/stderr/exit_code only | Artifacts (file output) deferred to v2 |
| Feature flag | `sandbox` (subprocess), `sandbox-docker` (Docker) | Entire module optional, keeps binary small |
| Docker integration | bollard crate (Rust SDK) | Better error handling than CLI shelling |

## Module Structure

```
src/
├── sandbox/
│   ├── mod.rs              # SandboxBackend trait, types, errors
│   ├── subprocess.rs       # SubprocessBackend (always compiled with feature)
│   ├── docker.rs           # DockerBackend (behind sandbox-docker feature)
│   └── firecracker.rs      # FirecrackerBackend (future, behind sandbox-firecracker)
└── nodes/
    └── sandbox.rs          # SandboxNode — implements Node trait
```

## Trait Definition

```rust
#[async_trait]
pub trait SandboxBackend: Send + Sync + 'static {
    /// Execute code in a sandboxed environment.
    async fn execute(&self, req: SandboxRequest) -> std::result::Result<SandboxResult, SandboxError>;

    /// Human-readable backend name for logs and metadata.
    fn name(&self) -> &str;

    /// Runtime check: is the backend usable on this system?
    fn available(&self) -> bool;
}
```

## Types

```rust
pub struct SandboxRequest {
    pub runtime: String,              // "python3", "node", "bash"
    pub code: String,
    pub env: HashMap<String, String>,
    pub timeout: Duration,
    pub memory_mb: Option<u64>,
    pub network: bool,                // default: false
    pub readonly_fs: bool,            // default: true (except /tmp)
    pub max_output_bytes: u64,        // cap stdout/stderr, default 1MB
}

pub struct SandboxResult {
    pub stdout: String,
    pub stderr: String,
    pub exit_code: i32,
    pub duration_ms: u64,
}

pub enum SandboxError {
    Timeout { timeout_seconds: u64 },
    RuntimeNotFound { runtime: String },
    ExitCode { code: i32, stderr: String },
    BackendUnavailable { backend: String, reason: String },
    ResourceLimit { resource: String, limit: String },
    Io(std::io::Error),
}
```

## Backend Implementations

### SubprocessBackend (feature: `sandbox`)

- Spawns `tokio::process::Command` with `kill_on_drop(true)`
- Wraps with `tokio::time::timeout`
- Writes code to temp file, runs via runtime binary (`python3`, `node`, `bash`)
- Collects stdout/stderr, truncates at `max_output_bytes`
- **No real isolation** — logs warning at startup
- Available on macOS, Linux, Windows

### DockerBackend (feature: `sandbox-docker`)

- Uses `bollard` crate to communicate with Docker daemon
- Creates container with resource limits:
  - `--memory` from `memory_mb`
  - `--cpus` from config
  - `--network none` when `network: false`
  - `--pids-limit` to prevent fork bombs
  - Read-only root filesystem, writable `/tmp` only
- Mounts code as a file inside container
- Pre-configured images per runtime (configurable in r8r.toml)
- `available()` pings Docker daemon

### FirecrackerBackend (feature: `sandbox-firecracker`, future)

- REST API over Unix socket to Firecracker VMM
- Snapshot/restore for warm pools
- Linux + KVM only, `available()` checks `/dev/kvm`
- Strongest isolation for multi-tenant deployments

## Feature Flags

```toml
[features]
default = []
redis = ["dep:redis"]
sandbox = []                                  # subprocess backend
sandbox-docker = ["sandbox", "dep:bollard"]   # implies sandbox

[dependencies]
bollard = { version = "0.18", optional = true }
```

Workflows using `type: sandbox` without the feature compiled get:
`"Sandbox node not available — rebuild with --features sandbox"`

## SandboxNode (Workflow Node)

```rust
pub struct SandboxNode {
    backend: Arc<dyn SandboxBackend>,
}

impl SandboxNode {
    pub fn new(backend: Arc<dyn SandboxBackend>) -> Self {
        Self { backend }
    }
}

#[async_trait]
impl Node for SandboxNode {
    fn node_type(&self) -> &str { "sandbox" }
    fn description(&self) -> &str {
        "Execute code in a sandboxed environment (Python, Node, Bash)"
    }
    async fn execute(&self, config: &Value, ctx: &NodeContext) -> Result<NodeResult> {
        // 1. Parse SandboxNodeConfig from config Value
        // 2. Render code template with ctx (render_template)
        // 3. Build SandboxRequest (apply global security caps)
        // 4. Call self.backend.execute(request)
        // 5. Try to parse stdout as JSON for structured output
        // 6. Return NodeResult with data + metadata
    }
}
```

Registered in `main.rs` after backend selection (not in `NodeRegistry::new()`):

```rust
let mut registry = NodeRegistry::new();

#[cfg(feature = "sandbox")]
{
    let sandbox_backend = build_sandbox_backend(&config);
    registry.register(Arc::new(SandboxNode::new(sandbox_backend)));
}
```

## Workflow YAML Interface

```yaml
nodes:
  - id: analyze_data
    type: sandbox
    config:
      runtime: python3
      timeout_seconds: 30
      memory_mb: 256
      network: false
      env:
        DATA_URL: "{{ nodes.fetch.output.url }}"
      code: |
        import json
        data = json.loads('{{ input | tojson }}')
        result = {"total": sum(item["value"] for item in data)}
        print(json.dumps(result))
```

### Node Output

```json
{
  "data": {
    "stdout": "{\"total\": 42}",
    "stderr": "",
    "exit_code": 0,
    "output": {"total": 42}
  },
  "metadata": {
    "backend": "docker",
    "runtime": "python3",
    "duration_ms": 1234,
    "memory_mb": 256,
    "network": false
  }
}
```

`output` field: if stdout is valid JSON, parsed into a Value. Otherwise null. Downstream nodes access structured data via `{{ nodes.analyze_data.output.output.total }}`.

## Configuration

### r8r.toml

```toml
[sandbox]
backend = "subprocess"   # "subprocess" | "docker"

[sandbox.docker]
python_image = "python:3.12-slim"
node_image = "node:22-slim"
bash_image = "alpine:3.19"
default_memory_mb = 256
default_timeout_seconds = 30

[sandbox.security]
allow_network = false           # global kill switch
max_memory_mb = 512             # global cap
max_timeout_seconds = 300       # 5 min cap
max_output_bytes = 1048576      # 1MB
allowed_runtimes = ["python3", "node", "bash"]
```

## Security

### Enforcement Per Backend

| Constraint | Subprocess | Docker |
|---|---|---|
| Timeout | `tokio::time::timeout` + `kill_on_drop` | Container stop timeout + kill |
| Memory | Not enforced (OS limits) | `--memory` via bollard |
| Network | Not enforced (inherits host) | `--network none` |
| Filesystem | Temp dir only, cleaned after | Read-only root, writable `/tmp` |
| CPU | Not enforced | `--cpus` via bollard |
| Output cap | Truncate stdout/stderr | Truncate on read |
| Process count | Not enforced | `--pids-limit` |

### Security Rules

1. **Network off by default.** YAML must explicitly set `network: true`. Docker enforces with `--network none`. Subprocess logs warning.
2. **Credentials never auto-injected.** SandboxNode strips credential references. API keys must be passed explicitly via `env` in YAML.
3. **No host filesystem access.** Code runs in temp dir (subprocess) or isolated container (Docker).
4. **Output size capped.** Default 1MB. Prevents memory exhaustion from malicious scripts.
5. **Global security overrides.** `[sandbox.security]` caps in r8r.toml override any per-node config.
6. **Subprocess backend warns.** Logs at startup: "Subprocess sandbox provides no isolation. Use Docker backend for production."
7. **Runtime whitelist.** Only runtimes in `allowed_runtimes` can be used.

## Error Handling

`SandboxError` maps to `Error::Node(...)` at the SandboxNode boundary:

| SandboxError | User-facing message |
|---|---|
| `Timeout` | "Sandbox execution timed out after {N}s" |
| `RuntimeNotFound` | "Runtime '{name}' not found. Available: python3, node, bash" |
| `ExitCode` | "Code exited with status {code}: {stderr}" |
| `BackendUnavailable` | "Sandbox backend '{name}' unavailable: {reason}" |
| `ResourceLimit` | "Resource limit exceeded: {resource} ({limit})" |

## Testing Strategy

1. **Unit tests** — Each backend module tests request building, output parsing, error mapping
2. **Integration tests** — Run real Python/Node/Bash code via subprocess backend
3. **Docker tests** — `#[cfg(feature = "sandbox-docker")]` tests requiring Docker daemon
4. **SandboxNode tests** — MockSandboxBackend with canned responses
5. **Security tests** — Verify timeout kills process, output truncation, runtime whitelist

### MockSandboxBackend

```rust
#[cfg(test)]
pub struct MockSandboxBackend {
    pub response: SandboxResult,
}

#[async_trait]
impl SandboxBackend for MockSandboxBackend {
    async fn execute(&self, _req: SandboxRequest) -> Result<SandboxResult, SandboxError> {
        Ok(self.response.clone())
    }
    fn name(&self) -> &str { "mock" }
    fn available(&self) -> bool { true }
}
```

## v2 Roadmap (Out of Scope for v1)

- **File artifacts** — `SandboxArtifact` struct, output directory collection
- **Firecracker backend** — Linux + KVM, snapshot/restore warm pools
- **Package caching** — Pre-installed pip/npm packages in Docker images
- **Streaming output** — WebSocket streaming of stdout during execution
- **WASM backend** — For trusted, high-frequency tool execution
- **Warm pools** — Pre-started containers/VMs for sub-100ms cold starts
